# ADR-0013: Model Logic Grid as anchor-aligned bijections

## Status

Accepted

## Date

2026-07-19

## Owner

Repository owner

## Context

M3 requires an immutable Logic Grid specification for unordered one-to-one associations across three or more categories. The representation must preserve user-facing item identity, support categorical and ordered clues, project naturally into a triangular grid, and reuse the common puzzle, expression, clue, and provenance contracts.

A pairwise Boolean matrix is visually direct but duplicates association truth across category pairs and needs additional closure constraints. Mapping each item to an opaque internal integer is compact, but it exposes solver-specific codes and leaves row identity ambiguous. A family model unrelated to `PuzzleSpec` would duplicate canonical identity, serialization, and provenance.

## Decision

Represent every category as a common `Domain` and every item as one aligned `Variable`. A `LogicGridCategory` binds the category domain to its ordered tuple of item variables. One declared anchor category supplies the assignment domain for every item variable.

Anchor-category variables are structurally assigned to their corresponding anchor values. Every category has exactly one all-different constraint, so its items form a bijection over the anchor rows. Equality between two item variables means the items are associated; inequality means they are not. Ordered, numeric-difference, alternative, cardinality, implication, and compound clues use the safe common expression tree and must close over declared variables.

The anchor category, category order, item order, and structural assignments are canonical specification data. They are not chosen privately by a solver. Clue text remains presentation data; normalized constraints remain authoritative and retain primary clue provenance.

## Alternatives

- Store one Boolean cell for every category pair. Rejected because equivalent associations would be repeated and require transitive consistency machinery in the canonical model.
- Give every item a hidden row number generated by a backend. Rejected because canonical identities and explanations must not depend on opaque solver codes.
- Model only assignments from one category into every other category. Rejected because non-anchor item relationships and future cross-category clue explanations become indirect and asymmetric.
- Create an unrelated Logic Grid schema. Rejected because it would bypass the common immutable puzzle and provenance contracts.

## Consequences

The representation has one stable variable per visible item and one stable row per anchor value. Grid cells are derived comparisons rather than duplicated source-of-truth facts. Category bijections and anchor symmetry breaking are validated before reasoning or solver integration.

An ordered or numeric clue can use direct comparison or difference expressions only when the chosen anchor domain declares the required semantics. Parsers, builders, and generators must therefore select the anchor deliberately and preserve it during canonical serialization.

This packet adds no runtime dependency, solver encoding, human rule, content, delivery surface, or deployment boundary.

## Reconsideration triggers

Revisit this decision if independent Logic Grid encodings cannot preserve these semantics, if user interaction cannot derive cells efficiently from item assignments, or if the ordered-association family demonstrates that anchor alignment cannot represent its required clues without family-specific bypasses.
